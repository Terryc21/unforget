## Scope decision (from 2026-05-02 planning conversation)

**v0.2 ships** plugin packaging + safety + must-ship Tier 6 fixes only. **v0.3 holds** the new subcommands (`/unforget close`, `/unforget delete`) and the permanent design decisions (External Target value, Hidden Status value, archive policy).

Why this split:
- Plugin packaging is the most user-visible change. Shipping it alone, with critical safety + adoption-path fixes, gets adopters using the plugin form. Anything beyond that is bonus.
- Tier 4 features are non-trivial design work. Holding them lets v0.2 ship in weeks, not months.
- Tier 3 design decisions need beta feedback before locking in. Better to ship v0.2 to a small audience, get feedback, then commit Tier 3 in v0.3.
- v0.2 can validate the plugin packaging path itself, separate from validating new features.

**v0.2 ship criteria:**
1. Plugin packaging works. Adopters install via Claude Code plugin manager (or git-clone-then-install pattern, whichever the platform supports). Invocation drops the `/skill` prefix; users type `/unforget init` directly.
2. All Tier 1 safety items remain in place (header notice, format-version marker, recovery section, example file).
3. Tier 2 recoverability lands (R1 dry-run, R2 backup, R3 format-version detection on read).
4. Must-ship Tier 6 surface-survey fixes land (T-2 filename archive exclusion, T-5 radar-suite format compatibility, T-6 .agents/plans path, T-10 general doc scanning, T-11 roadmap-file detection).
5. The Stuffolio fresh-init re-test produces ≥70% of expected candidates (target: 42 of 60 rows). v0.1 baseline was ~9 of 60.
6. README updated to reflect plugin install + invocation form. Version bumped to 0.2.0.
7. SKILL.md / plugin manifest version-stamped.

## Phase 1 — Foundation: plugin packaging

**Goal:** Move from SKILL.md-only to Claude Code plugin (`.claude-plugin/`), with no behavior changes from v0.1.x.

**Why first:** all subsequent phases edit the spec content. If plugin packaging is left until last, we have to retest every Tier 6 change twice (once in SKILL.md form, once in plugin form). Doing packaging first means every later change is tested in its final form.

### Tasks

1. **Reference an existing Claude Code plugin** to copy structure from. radar-suite has been mentioned as plugin-based in user discussions; check its `.claude-plugin/` layout.
2. **Create plugin scaffold:**
   - `.claude-plugin/manifest.json` (or .toml depending on convention) with name, version, description, slash commands.
   - Slash command bindings for the 7 v0.1 subcommands (`init`, `add`, `edit`, `import`, `list`, `scan`, `promote`).
   - Plugin entry point that loads SKILL.md content (or replaces it with plugin-resident files).
3. **Verify invocation:** install the plugin locally, type `/unforget` in a fresh Claude Code session, confirm it responds. The `/skill` prefix should no longer be required.
4. **Verify backward compat:** running v0.2 against Stuffolio's existing v1-format UNFORGET.md should produce no errors. The format-version marker (R3) handles this in Phase 2; for now, just confirm no crashes.
5. **Update install docs:** README "Install" section becomes plugin-install instructions. Manual install path (the `cp SKILL.md` fallback) preserved as secondary option.
6. **Commit:** "v0.2 Phase 1: plugin packaging, no behavior changes"

### Open questions for Phase 1

- Does the Claude Code plugin format support multiple slash commands per plugin, or is each subcommand a separate plugin? Affects manifest structure.
- Does the plugin load SKILL.md as content, or replace it with plugin-native command files? If the latter, SKILL.md becomes documentation only and the spec lives in command-handler files.
- How does v0.2 plugin coexist with v0.1 SKILL.md install if both are present in `~/.claude/skills/unforget/`? Document the migration path.

### Phase 1 ship gate

`/unforget --version` (D4) returns `0.2.0`. `/unforget init` runs without crashing on a clean test project. Backward-compat test: `/unforget list` works on Stuffolio's existing UNFORGET.md without errors.

## Phase 2 — Tier 2 recoverability

**Goal:** Make destructive operations safe to refine. R1, R2, R3 are prerequisites for Phase 3 confidence.

### Tasks

1. **R1 — `--dry-run` flag on `promote`:**
   - Add to spec: `/unforget promote --dry-run` shows the diff without applying.
   - Implementation: capture the planned changes, render them as a markdown diff, prompt user "apply?" before write.
   - Test: run on Stuffolio's UNFORGET.md scratch copy with --dry-run; confirm no file changes.
2. **R2 — Auto-backup before destructive operations:**
   - Spec: before `promote` writes, create `UNFORGET.md.bak-YYYY-MM-DD-HHMMSS` in the same directory.
   - Keep last 5 backups; delete older ones automatically.
   - Add `*.bak-*` to a `.gitignore` template the init flow offers to apply.
   - Test: run promote twice in a day; confirm two backups present.
3. **R3 — Format-version detection on every read:**
   - Spec: `add` / `list` / `promote` / `scan` / `edit` open the file, check for `<!-- unforget-format: vN -->` marker.
   - If marker absent: prompt user "this file may not be in unforget format; proceed anyway?" Default no.
   - If marker recognized (v1): proceed normally.
   - If marker is a future version (v2+ when the skill is v0.2 reading v1): proceed read-only with warning, refuse writes.
   - Test: synthesize a v0.5 marker file, confirm v0.2 refuses to write to it.

### Phase 2 ship gate

`/unforget promote --dry-run` produces a clean diff. Backups appear after `promote` runs. A v0.1 file (no format-version marker) prompts the user and proceeds with consent. A future-version marker file is refused.

## Phase 3 — Critical Tier 6 fixes

**Goal:** Fix the surface-survey gaps that prevent fresh init from finding real deferred work. After this phase, the Stuffolio re-test should jump from ~9 candidates to ~30+.

### Tasks

1. **T-2 — Filename archive exclusion:**
   - Spec change in Phase 2 universal exclusion rule: also skip files matching `*archive*.md` (case insensitive) regardless of path.
   - Test: run scan against scratch Stuffolio; confirm `Deferred-archive-2026-05-01.md` is skipped.
2. **T-5 — radar-suite format compatibility:**
   - Decide: pin radar-suite version + update spec to v3 schema, OR use scratch reports (`scratch/*-radar-*.md`) as the deferral source.
   - Recommend: do both. Pin radar-suite v3+ as the supported format. Use scratch reports as the per-finding source. Use ledger.yaml only for "was this audit run?" metadata.
   - Update SKILL.md Phase 2 Surface 2 description.
   - Test: run scan against scratch Stuffolio; confirm radar-suite findings are extracted.
3. **T-6 — Add `.agents/plans/` to default scan paths:**
   - Spec change: Phase 2 Surface 3 default scan paths become `./plans/`, `./.claude/plans/`, `./.agents/plans/`. Generalize to `**/plans/*.md` excluding archive paths.
   - Test: run scan against scratch Stuffolio; confirm 4 plan files in `.agents/plans/` are found.
4. **T-10 — General doc scanning:**
   - Spec addition: new Surface 1b. Scan `Documentation/Development/*.md`, `Documentation/Notes/*.md`, `scratch/*.md`, and equivalents for files whose content shape suggests deferral artifacts.
   - Heuristics: heading text matching "Deferred", "Pending", "TODO", "Phase N pending"; date-prefix file names; explicit "DEFERRED:" prefixes in headings.
   - Each match prompts user for import decision (don't auto-import; prose-shape detection is fuzzy).
   - Test: run scan against scratch Stuffolio; confirm `2026-04-30-test-suite-failures.md` in `Documentation/Development/` is offered as a candidate.
5. **T-11 — Recognize roadmap-shaped files:**
   - Spec change: Surface 1 regex extended to also match `*roadmap*.md`, `ROADMAP.md`, `*plan*.md` (case insensitive).
   - Plus content-shape heuristic: 3+ headings matching "Tier N" / "Phase N" / "Priority N" patterns flag a file as a deferral artifact.
   - Test: run scan against the unforget public repo itself; confirm `v0.2-roadmap.md` and this file (`v0.2-release-plan.md`) are both surfaced.

### Phase 3 ship gate

Stuffolio re-test produces ≥30 candidates (vs ~9 baseline). The unforget self-test (init run on the unforget repo) finds at least the v0.2 roadmap and release plan.

## Phase 4 — High-priority Tier 6 polish

**Goal:** Fix the remaining surface-survey gaps. Less critical than Phase 3 but each addresses a real adopter pain point.

### Tasks

1. **T-3 — Detect redirect-pointer files:**
   - Heuristic: short files (< 30 lines) containing pointer phrases ("MOVED", "see also", path references to UNFORGET.md or another deferral file).
   - Skip with status note rather than parse.
2. **T-4 — Hardcode known files within audit-tool dirs:**
   - Phase 2 Surface 2 spec change: only specific files within `.radar-suite/` are deferral sources. Currently: `ledger.yaml` plus `scratch/*-radar-*.md`. Other yaml in `.radar-suite/` is skipped.
3. **T-7 — Fourth state for `gh` surface:**
   - Detect missing `.git/` or non-GitHub remote before invoking `gh`.
   - Skip with: "GitHub issues skipped (project not on GitHub)."
4. **T-8 — Memory directory detection per AI tool:**
   - From Phase 1 wiring step (which AI file was wired), determine memory path.
   - Claude Code → `~/.claude/projects/<encoded>/memory/`.
   - Cursor → `~/.cursor/memory/` or equivalent (research the actual path).
   - Aider → `.aider/` or equivalent.
   - If multiple AI tools wired: scan each.
   - Document the path mapping explicitly in SKILL.md.
5. **T-9 — Content-shape check for memory meta-files:**
   - Before importing a memory file, scan for meta-doc phrases ("single source of truth", "format:", "how to use").
   - If detected: flag for user decision rather than auto-import.

### Phase 4 ship gate

Stuffolio re-test produces ≥42 candidates (≥70% of the ~60-row target). All remaining missed candidates have a clear explanation (tacit knowledge that no scan can find).

## Phase 5 — Documentation + ship readiness

**Goal:** Documentation is complete. Skill is releasable.

### Tasks

1. **D1 — Recommended file path in README + init flow:**
   - README "Install" / "Quick start" recommends `Documentation/Development/Deferred/UNFORGET.md` as default.
   - Init flow Phase 1 already does this, per spec.
2. **D2 — Auto-fill defaults table in SKILL.md:**
   - Currently the init flow says "conservative defaults" without specifying. Pin the values:
     - Target: `🌫️ SOMEDAY`
     - Urgency: `🟢 MEDIUM` unless source says CRITICAL/HIGH/LOW explicitly
     - Risk: Fix: `⚪ Low`
     - Risk: No Fix: `⚪ Low` unless source mentions data loss/crash/user-visible
     - ROI: `🟢 Good`
     - Blast Radius: `⚪ 1 file` unless source mentions count
     - Fix Effort: `Small`
     - Status: `Open`
3. **D3 — Re-running init policy:**
   - If `init` runs against an existing UNFORGET.md, abort with: "UNFORGET.md already exists. Use `/unforget import` to add new items, or delete the file if you want to start over."
4. **D4 — Install verification command:**
   - `/unforget --version` prints version + install path + format-version supported.
5. **D5 — Format reference in SKILL.md:**
   - Detail-block format (closure pointer / body / spawn links) currently lives only in `examples/UNFORGET.md`. Move it into SKILL.md as a first-class section.
6. **README updates for v0.2:**
   - Install instructions reflect plugin packaging.
   - Invocation form: `/unforget` not `/skill unforget`.
   - "Beta testers wanted" call-out re-added (was removed in v0.1.x cleanup) — v0.2 is the natural moment to invite real-project beta users.
   - v0.2-roadmap.md status fields updated: shipped items move to "Implemented", deferred items move to v0.3 roadmap.
7. **Version bump:**
   - `SKILL.md` frontmatter `version: 0.2.0`
   - Plugin manifest version: 0.2.0
   - Git tag: `v0.2.0`
8. **Push, announce:**
   - Push to GitHub.
   - Tag and create release notes.
   - Announce: GitHub release page, Reddit r/ClaudeAI post, mention in any Stuffolio session work.

### Phase 5 ship gate

`/unforget --version` returns 0.2.0. Stuffolio re-test passes the ≥70% threshold from Phase 4. README accurately describes v0.2 install and invocation. All Phase 1-5 commits are pushed and tagged.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Plugin packaging (Phase 1) blocks on Claude Code platform docs we haven't read yet | Open question; resolve early. If plugin format is not yet documented or stable, fall back to "v0.2 ships as a polished SKILL.md with the spec fixes; v0.3 ships plugin packaging." |
| T-5 (radar-suite format compatibility) requires understanding radar-suite v3 schema in detail | Read radar-suite repo's own docs before Phase 3. radar-suite is at `/Volumes/2 TB Drive/Coding/GitHub/radar-suite/`; spec lives in its own SKILL.md. |
| Tier 6 fixes might conflict with Tier 2-4 design decisions deferred to v0.3 | Phase ordering helps: do Tier 2 (recoverability) before Tier 6 (which is destructive in some cases). Don't lock Tier 3 design decisions until v0.3 beta. |
| Stuffolio's scratch copy goes stale between tests | Keep the scratch dir until v0.2 ships. If Stuffolio's UNFORGET.md gets restructured, refresh the scratch copy. |
| v0.2 release timeline slips past Stuffolio's build 33 ship | Stuffolio doesn't depend on v0.2. The Stuffolio side is hand-maintained; v0.2 is for new adopters. No coupling. |
| Beta testers can't be recruited in time | v0.2 can ship without beta validation. Tier 3 design decisions stay deferred to v0.3 specifically because they need beta feedback. v0.2 itself is a low-risk safety + adoption-path improvement. |

## How to use this plan

1. Review this file. Adjust phases, ship gates, or scope before starting Phase 1.

5. After v0.2 ships: archive this file (rename to `v0.2-release-plan-archive.md` or move to `releases/`). Start `v0.3-release-plan.md` for the next cycle.
