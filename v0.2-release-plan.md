## Scope decision (from 2026-05-02 planning conversation)

**v0.2 ships** plugin packaging + safety + must-ship Tier 6 fixes only. **v0.3 holds** the new subcommands (`/unforget close`, `/unforget delete`) and the permanent design decisions (External Target value, Hidden Status value, archive policy).

Why this split:
- Plugin packaging is the most user-visible change. Shipping it alone, with critical safety + adoption-path fixes, gets adopters using the plugin form. Anything beyond that is bonus.
- Tier 4 features are non-trivial design work. Holding them lets v0.2 ship in weeks, not months.
- Tier 3 design decisions need beta feedback before locking in. Better to ship v0.2 to a small audience, get feedback, then commit Tier 3 in v0.3.
- v0.2 can validate the plugin packaging path itself, separate from validating new features.

**v0.2 ship criteria:**
1. Plugin packaging works. Adopters install via Claude Code plugin manager (or git-clone-then-install pattern, whichever the platform supports). Invocation drops the `/skill` prefix; users type `/unforget init` directly.
2. All Tier 1 safety items remain in place (header notice, format-version marker, recovery section, example file).
3. Tier 2 recoverability lands (R1 dry-run, R2 backup, R3 format-version detection on read).
4. Must-ship Tier 6 surface-survey fixes land (T-2 filename archive exclusion, T-5 radar-suite format compatibility, T-6 .agents/plans path, T-10 general doc scanning, T-11 roadmap-file detection).
5. The Stuffolio fresh-init re-test produces ≥70% of expected candidates (target: 42 of 60 rows). v0.1 baseline was ~9 of 60.
6. README updated to reflect plugin install + invocation form. Version bumped to 0.2.0.
7. SKILL.md / plugin manifest version-stamped.

### Open questions for Phase 1 — resolved 2026-05-02

- **Q: Does the Claude Code plugin format support multiple slash commands per plugin, or is each subcommand a separate plugin?**
  A: One plugin can declare multiple skills (radar-suite has 8). Subcommands within a single skill (e.g., `/unforget init`, `/unforget add`) are handled by the skill itself, not split into separate plugin entries. unforget ships as one plugin with one skill that handles all 7 subcommands internally.
- **Q: Does the plugin load SKILL.md as content, or replace it with plugin-native command files?**
  A: Plugin loads SKILL.md as content (verified by working radar-suite layout and confirmed by live test — `/unforget` invocation triggered the existing SKILL.md spec). No native command files needed; SKILL.md remains the spec.
- **Q: How does v0.2 plugin coexist with v0.1 SKILL.md install if both are present in `~/.claude/skills/unforget/`?**
  A: Not tested directly. The v0.1 fallback documentation now says "either install path works, but don't run both at once." If a user has both, Claude Code's plugin manager should win (loaded earlier in the discovery order), but this is environment-dependent. Document as a caveat rather than fix programmatically.
