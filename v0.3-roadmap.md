## New for v0.3 (proposals from v0.2 retrospective)

These came out of a 2026-05-02 conversation about whether `/schedule` (Claude Code's remote-agent scheduling skill) should be integrated into unforget. The conclusion was no: integration would break the "any editor, any AI, any project" portability that makes unforget's pitch work, would couple adopters to Claude Code, and would create invisible failure modes when external scheduling breaks. But the underlying need (time-bound deferrals and external scheduling) is real. The two items below address that need without coupling unforget to any scheduler.

### Rating

| # | Finding | Urgency | Risk: Fix | Risk: No Fix | ROI | Blast Radius | Fix Effort | Status |
|---|---|---|---|---|---|---|---|---|
| 1 | T1 `Wake-up` field for time-bound deferrals (format addition) | 🟢 MEDIUM | ⚪ Low | 🟢 Medium | 🟠 Excellent | ⚪ 1 file | Small | Proposal |
| 2 | T2 README "Recipes" section: scheduling unforget scans externally | 🟢 MEDIUM | ⚪ Low | 🟢 Medium | 🟢 Good | ⚪ 1 file | Small | Proposal |

Risk: No Fix is Medium for both because adopters who want time-bound deferrals or push-style reminders are forced to invent their own conventions today, which fragments the format across projects. Adding both keeps unforget the universal format while making time-bound work first-class.

### T1: `Wake-up` field for time-bound deferrals

**Status:** Proposal

**Problem.** Today, a row that's dead until a future date (vendor reapplication window opens 2026-08-01, soak period ends after build 35, partner integration blocked until contract renewal) has no clean place to record the wake-up date. Users put it in the Finding cell as prose, in a footnote, or in a memory file that disappears at session end. None of those surfaces are queryable.

**Proposed shape.** An optional `Wake-up` field for rows that are time-bound. Two presentation options:

**Option A: detail-block field (recommended).** Lives in the detail block beneath the row, leaves the 10-column rating table unchanged.

```markdown
<details>
<summary>12 · Re-evaluate Best Buy API · 🌫️ SOMEDAY</summary>

**Wake-up:** 2026-08-01 (reapplication window opens)
**Closure pointer:** when reapplied, log the new ticket here.

The original application was rejected 2026-04-27. Reopen window closes 2026-05-07
(handled separately in the row above) but the next reapplication window starts 2026-08-01.
</details>
```

**Option B: 11th column.** Adds a `Wake-up` column to the rating table. Rejected for the same reason custom column reordering is rejected in `## Anti-patterns`: it expands the universal format and breaks comparability across projects that don't use it.

Recommend Option A. It's opt-in, doesn't bloat the table, and rows without a wake-up date are completely unaffected.

**Behavioral hook.** `/unforget scan` gains an awareness of the field:
- Rows with `Wake-up:` set to a date in the past are flagged as "woken" (default Recommendation: `investigate`).
- Rows with `Wake-up:` set to a future date are skipped from staleness checks (the row isn't stale; it's intentionally dormant).
- `/unforget scan --woken` shows only rows whose Wake-up date has passed, as a focused view for the weekly "what woke up this week?" ritual.

**What this does NOT do.** It does NOT fire any alert, schedule any agent, or push any notification. The Wake-up field is data the user reads when they choose to. External schedulers (see T2) handle the push side.

**Why this is the right shape for the public skill.** It's pure format work. Any AI tool that reads UNFORGET.md (Cursor, Aider, GitHub Copilot reading project docs) sees a date in a labeled field; it's parseable without unforget-specific tooling. No scheduler dependency, no Claude Code lock-in.

**Open questions for v0.3 design.**
1. Date format: ISO 8601 (`2026-08-01`) only, or also accept relative phrases (`+90 days`, `after build 35`)? Recommend ISO 8601 only for v0.3. Relative phrases are ambiguous without a reference date and harder for non-Claude AI tools to interpret.
2. Multiple wake-ups: can a row have a list of dates (e.g., quarterly check-ins)? Recommend single date only for v0.3. Recurring check-ins should be expressed as a `/schedule` cron externally, not in the row.
3. What if a wake-up date passes and the user doesn't act? The row stays in its current Target (typically `🌫️ SOMEDAY`); scan keeps surfacing it as woken until the user resolves it.

### T2: README "Recipes" section: scheduling unforget scans externally

**Status:** Proposal

**Problem.** Adopters routinely ask "how do I get reminded about deferred work without having to remember to run `/unforget scan`?" Today the README's only answer is the one-line note in `## Scheduling` ("`/unforget scan` is safe to run on demand. For automated recurring scans, the user can schedule it via Claude Code's `/schedule` skill or any cron-like external scheduler"). That sentence is correct but doesn't show the user how. Adopters interpret the silence as "you're on your own" and either build something fragile or stop thinking about it.

**Proposed shape.** Add a `## Recipes` section to README.md with worked examples for the most common external schedulers. Each recipe is a short paragraph plus a code block. The section should be neutral about which scheduler is best; the message is "pick whichever you already trust."

**Recipes to include in v0.3.**

1. **Claude Code `/schedule` (one-time wake-up).** For T1-style time-bound deferrals: `/schedule "run /unforget scan --woken and report rows that woke up" run_once_at: 2026-08-01T17:00:00Z`. Code-block example. Note that this requires Claude Code with `/schedule` configured.

2. **Claude Code `/schedule` (recurring weekly scan).** `/schedule "run /unforget scan and email me anything stale" cron: 0 14 * * 1` for Monday 2pm. Same caveats.

3. **GitHub Actions (recurring scan posted as issue comment).** A `.github/workflows/unforget-scan.yml` snippet that runs weekly, invokes the scan via Claude API or a CLI wrapper, and posts the result to a tracking issue. The wrapper does not exist yet (open question below); for v0.3 the recipe shows the workflow shape and links to the API call form.

4. **Plain cron + script.** A bash one-liner the user can drop into `~/Library/LaunchAgents/` (macOS) or `crontab` (Linux) that runs `claude-code` headless against the project, captures output, and emails or files-it-somewhere. This is the maximum-portability recipe.

5. **Linear / Jira due dates.** Not a scheduler per se, but the most common pattern in practice: when adding a row with a Wake-up date in unforget, also create a Linear issue with the same due date. The user reviews their Linear inbox weekly and the row resurfaces there. Recipe is a workflow note, not a code block.

**What this section deliberately does NOT do.** It does NOT recommend one scheduler. It does NOT integrate any of them into unforget. It does NOT create a hidden dependency. The Recipes section is pure documentation; deleting it would not break any installed unforget. It exists to show adopters what the external-scheduling pattern looks like, then get out of the way.

**Why this is the right shape for the public skill.** It strengthens the "unforget is the format, use any scheduler you want" pitch by making that statement actionable. Adopters bounce off "use any external scheduler" when no example is given; they adopt confidently when they see five working examples for the schedulers they already use.

**Open questions for v0.3 design.**
1. Does a `claude-code --headless --skill unforget --command scan` CLI form exist? If not, recipe 3 (GitHub Actions) and recipe 4 (plain cron) need a different invocation path. Confirm before drafting the recipes for real. If headless invocation isn't possible yet, recipes 3 and 4 reduce to "open Claude Code manually on a recurring reminder" which is much weaker.
2. Should the Recipes section live in README.md or in a separate `docs/RECIPES.md`? README is more discoverable; separate file keeps README short. Recommend README inline for v0.3 (3-5 recipes is small enough), promote to separate file in v0.4 if the section grows past 8 recipes.

Initial proposal (subject to revision when v0.3 release plan is written):
