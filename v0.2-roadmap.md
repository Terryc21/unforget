# unforget v0.2 Roadmap

This document tracks proposed changes for the v0.2 release. Items here are **proposals**, not committed spec — the v0.2 plugin packaging is the natural moment to lock in design decisions, and writing them up here first lets beta users react before they're permanent.

Status legend: `Proposal` (drafted, not decided) | `Approved` (committed for v0.2) | `Implemented` (in plugin code) | `Deferred` (post-v0.2)

---

## Source of these proposals

These items came out of using the v0.1 skill against a real shipping project (Stuffolio, ~50-row UNFORGET.md, App Store submission cycles) for several weeks. Most are gaps that only show up at production scale; the v0.1 spec was validated against smaller files. The header notice and format-version marker (already shipped to the public README + example file) were the immediate-safety subset of this list.

The Tier 6 items below came from a separate exercise: simulating a fresh `/unforget init` against a scratch copy of Stuffolio with the live UNFORGET.md removed. That test surfaced 9 distinct ways the v0.1 surface-survey misses real deferred work in a complex project. Full report: [`test-reports/2026-05-02-stuffolio-fresh-init-simulation.md`](test-reports/2026-05-02-stuffolio-fresh-init-simulation.md).

## Tier 2 — Recoverability (proposals for v0.2)

| ID | Item | Status | Notes |
|---|---|---|---|
| R1 | `--dry-run` flag on `promote` and any other write command | Proposal | Show the diff that would be written without committing. Reduces fear of running `promote`. |
| R2 | Auto-backup before destructive operations | Proposal | Write `UNFORGET.md.bak-YYYY-MM-DD-HHMMSS` before `promote`. Keep last 5 by default; configurable. Tradeoff: backup files clutter the directory; need to add to `.gitignore` template by default. |
| R3 | Format-version detection on every read | Proposal | When `add` / `list` / `promote` / `scan` open the file, check the format-version marker. If absent or unrecognized, abort with a clear message rather than silently corrupting. |

## Tier 3 — Format clarifications (proposals for v0.2)

| ID | Item | Status | Notes |
|---|---|---|---|
| F1 | Archive policy for closed rows at promotion time | Proposal | Recommended: at `promote`, move all `Fixed / Skipped` rows older than the previous release to `UNFORGET-archive-YYYY-MM.md` in the same directory. Main file stays scannable; archive file preserves history. Tradeoff: cross-references from plans/commits to archived rows have to follow the archive trail. Mitigated by archive filename being predictable. |
| F2 | `Hidden` Status value (or convention `Fixed (workaround)`) | Proposal | Today, hiding a feature behind a flag and marking the row Fixed is misleading: a future reader can't tell the underlying functionality was actually fixed. Two options: add `Hidden` to the Status enum (cleanest but expands the enum), or document a convention `Fixed (workaround)` as text in the Status cell (no enum change but harder to filter on programmatically). Recommend the enum addition. |
| F3 | `External` Target value for items waiting on third parties | Proposal | Items blocked on vendor approvals, regulatory reviews, partner integrations are common. Folding them into "deferred" loses information. Adds a 5th Target value: `⏳ EXTERNAL`. Tradeoff: vocabulary expansion is permanent — every preset mode, every example, every doc has to know it. |
| F4 | Pre-release ship gate as auto-derived view, not hand-maintained | Proposal | Stuffolio's UNFORGET.md hand-maintains a "Pre-release ship gate" table at the bottom that duplicates THIS-target rows from the main tables. Sync drift is invisible. v0.2's `/unforget list --target=THIS` already provides this view; spec should document it as the canonical answer to "can we ship?" and discourage hand-maintained ship-gate tables. |

## Tier 4 — Feature additions (proposals for v0.2)

| ID | Item | Status | Notes |
|---|---|---|---|
| C1 | `/unforget close <ID>` subcommand | Proposal | Today closure is hand-edit. New command prompts for closure type (real fix / workaround / won't fix / superseded), asks if a follow-up row should be spawned, writes the closure pointer at the top of the detail block, updates Status, optionally creates the spawned row. ~5 seconds vs current "find row, find detail, edit two places, hope you don't miss the ship-gate table." |
| C2 | `/unforget delete <ID>` subcommand | Proposal | For rows that turned out invalid (duplicate, mistaken capture). Confirms before write, dry-run by default, retains the ID as "Retired" so cross-references don't silently point at a different row later. |
| C3 | Spawn-link automation | Proposal | When `close` spawns a follow-up, both rows get reciprocal `Spawned-from:` / `Spawns:` lines automatically. `add` / `edit` / `delete` maintain the cross-references. Manual chain maintenance rots; automation prevents it. |
| C4 | `/unforget link <ID-from> <ID-to>` for retroactive chain creation | Proposal | If a follow-up row was created without using `close`, this command links the pair after the fact. |

## Tier 5 — Documentation (proposals for v0.2 + ongoing)

| ID | Item | Status | Notes |
|---|---|---|---|
| D1 | Recommended file path called out in README + init flow | Proposal | Recommend `Documentation/Development/Deferred/UNFORGET.md` as default. Junior devs default to repo-root files, which is fine but inconsistent across projects. |
| D2 | Auto-fill defaults table in SKILL.md | Proposal | The init flow's "conservative defaults" are documented as a concept but never specified. Implementation will drift across versions. Spec should pin: Target=SOMEDAY, Urgency=MEDIUM, RFix=Low, RNo=Medium, ROI=Good, Blast=conservative-medium, Effort=Medium, Status=Open. |
| D3 | Re-running init policy | Proposal | Spec should say what happens if a user runs `init` twice. Recommended: detect existing UNFORGET.md, abort with a message pointing at `import` (which re-scans without re-creating). |
| D4 | Install verification command | Proposal | After install, user has no way to verify the skill loaded short of trying to use it. v0.2 plugin should support `/unforget --version` or similar that prints version + install path. |
| D5 | Format reference in SKILL.md | Proposal | Detail-block format (closure pointer / body / spawn links) needs to be in SKILL.md, not just the example file, so the skill's own docs include the full format. |

## Tier 6 — Surface survey gaps (from 2026-05-02 fresh-init simulation)

These are spec gaps in Phase 2 (surface survey) and Phase 4 (auto-fill) that surfaced when the v0.1 spec was simulated against a real complex project. Full test report: `test-reports/2026-05-02-stuffolio-fresh-init-simulation.md`.

| ID | Item | Severity | Status | Notes |
|---|---|---|---|---|
| T-1 | Multiple AI-instruction files detected and offered to user | n/a | Implemented (this works in v0.1) | Confirms Phase 1 Q3 spec is correct. No change needed. |
| T-2 | Universal exclusion rule extends to filename match | 🟡 HIGH | Proposal | Currently excludes `archive` path segments only. Should also exclude files matching `*archive*.md` (case insensitive) regardless of path. Stuffolio repro: `Documentation/Development/Deferred/Deferred-archive-2026-05-01.md` would be wrongly imported. |
| T-3 | Detect redirect-pointer files | 🟢 MEDIUM | Proposal | Short files (< 30 lines) containing pointer phrases ("MOVED", "see also", absolute or relative path references to UNFORGET.md or another deferral file) should be skipped with a status note, not parsed as deferral entries. Stuffolio repro: root `Deferred.md` is a 12-line redirect to UNFORGET.md. |
| T-4 | Hardcode known files within audit-tool directories | 🟡 HIGH | Proposal | `.radar-suite/` has 7 yaml files; only `ledger.yaml` is the deferred-work source. Spec must explicitly restrict scanning to known files per audit tool, not "any yaml under `.radar-suite/`". |
| T-5 | radar-suite ledger.yaml format compatibility | 🔴 CRITICAL | Proposal | SKILL.md describes radar-suite ledger format that doesn't match v3 reality. Either pin a version + update spec to match v3 schema, or use scratch reports (`scratch/*-radar-*-2026-MM-DD.md`) as the source instead. |
| T-6 | Add `.agents/plans/` to default scan paths | 🟡 HIGH | Proposal | Current spec scans `./plans/` and `./.claude/plans/` only. `.agents/plans/` is the convention used by radar-suite and other audit skills. Generalize to `**/plans/*.md` excluding archive paths. |
| T-7 | Fourth state for `gh` surface (project not on GitHub) | 🟢 MEDIUM | Proposal | Spec lists 3 states (gh missing / not authed / authed-empty); doesn't cover the "not a git repo" or "git remote isn't GitHub" case. Detect missing `.git/` or non-GitHub remote before invoking `gh`. |
| T-8 | Memory directory detection per AI tool | 🟡 HIGH | Proposal | Spec says "memory files match `^deferred_*.md`" without saying where memory lives. Detect from Phase 1 wiring step: Claude Code → `~/.claude/projects/<encoded>/memory/`; Cursor / Aider → different paths. Document the mapping. |
| T-9 | Content-shape check for memory meta-files | 🟢 MEDIUM | Proposal | Filename matching catches meta-docs like `deferred_work_index.md`. Add a quick content check ("single source of truth", "format:", "how to use") to flag those for user decision rather than auto-import. |
| T-10 | Expand Phase 2 to scan general documentation files for deferral artifacts | 🔴 CRITICAL | Proposal | Phase 2 surfaces miss ~80% of real deferred items in a complex project. Most missed rows live in `scratch/*.md` audit reports, `Documentation/Development/*.md` (e.g., date-prefixed planning files), and tacit knowledge. Add content-shape scanning for these locations: heading text matching "Deferred"/"Pending"/"TODO"/"Phase N pending", date-prefix file names, explicit "DEFERRED:" prefixes. For tacit knowledge, recommend Phase 6 deep-dump be promoted to default-on for first-time adoption. |
| T-11 | Recognize roadmap-shaped files in Surface 1 regex | 🔴 CRITICAL | Proposal | The unforget repo's own `v0.2-roadmap.md` is invisible to the skill. Surface 1 regex misses any file not literally named `Deferred.md` / `BACKLOG.md` / `TODO.md` / `*deferred*.md`. Add `*roadmap*.md`, `ROADMAP.md`, `*plan*.md` (case insensitive) to the regex. Also add a content-shape heuristic: 3+ headings matching "Tier N" / "Phase N" / "Priority N" patterns flag a file as a deferral artifact. Test report calls this a self-test failure: the skill that's supposed to surface deferred work can't surface its own. |
| T-12 | T-5 filename pattern too narrow for actual radar-suite scratch reports | 🔴 CRITICAL | Proposal (Phase 4) | Phase 3 T-5 spec'd `scratch/*-radar-*-YYYY-MM-DD.md` as the per-finding source. Stuffolio repro: `scratch/` contains ~80 files named `audit-<domain>-YYYY-MM-DD.md` (e.g., `audit-concurrency-2026-04-25.md`). None match `*-radar-*-`. Surface 2's per-finding source returns 0 hits against a project that has 80 active audit reports. Phase 4 fix belongs alongside T-4 (hardcode known files within audit-tool dirs): the spec must list the actual radar-suite v3+ filename pattern (`audit-<skill>-YYYY-MM-DD.md`). Source: `test-reports/2026-05-02-phase3-shipgate-retest.md`. |
| T-13 | T-10 heuristics miss audit-report severity-tier headings | 🔴 CRITICAL | Proposal (Phase 4) | Phase 3 T-10 spec'd six heuristics for Surface 1b: heading text matching `Deferred` / `Pending` / `TODO` / `Phase N pending`, date-prefix filenames, `DEFERRED:` heading prefixes. Stuffolio repro: audit reports use severity-tier headings (`## HIGH Issues`, `## MEDIUM Issues`) and per-finding ID headings (`### CONC-H1: <title>`, `### CONC-M1: <title>`). None match the six heuristics. Audit reports that are implicitly deferred (the audit was run, findings exist, no fix has shipped) don't surface. Phase 4 fix: add a 7th heuristic: heading text matching severity-tier prefixes (`HIGH Issues`, `MEDIUM Issues`, `LOW Issues`) or per-finding ID patterns (regex `^[A-Z]+-[A-Z][0-9]+:`). Source: `test-reports/2026-05-02-phase3-shipgate-retest.md`. |

### Tier 6 ship priority

Items T-2, T-5, T-6, T-10, T-11 are the must-ship subset for v0.2. Without them, fresh adopters running init against a complex project get a partial UNFORGET.md and misleading results. T-3, T-4, T-8 are second priority. T-7, T-9 are nice-to-have polish.

**Phase 3 retest (2026-05-02) added T-12, T-13.** Both are 🔴 CRITICAL because they keep the largest single coverage source (radar-suite scratch audit reports) effectively invisible. Recommend folding them into Phase 4 as must-ship alongside T-4. Without them, the v0.2 ship gate passes by the floor (30 candidates exact) but radar-suite users get a partial UNFORGET.md.

## Out of scope for v0.2

These came up during the review but aren't urgent enough or have unclear designs.

| Item | Why deferred |
|---|---|
| ID collision avoidance between AI assistants writing to UNFORGET.md concurrently | Multi-skill coordination is a future problem; rare in practice. |
| Auto-commit hook hardening (prettier / markdownlint compatibility) | Address reactively when reports come in. |
| Stale-prompt protection (continuation prompts pointing at outdated tracking files) | Session-level discipline issue, not a skill issue. |
| Per-row column visibility | Ruled out (anti-pattern). |
| Custom column reordering | Ruled out (anti-pattern). |
| Multiple-file UNFORGET.md splits | Ruled out (anti-pattern). |

## How to comment on proposals

If you're a v0.1 user and have an opinion on any of the above, [open an issue](https://github.com/Terryc21/unforget/issues) referencing the item ID (e.g., "F3 - External Target value"). Especially valuable: opinions backed by real-project usage rather than hypothetical concerns.

The v0.2 ship target is when these proposals lock in. Anything not Approved by then ships in a later version.
