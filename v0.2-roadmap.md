# unforget v0.2 Roadmap

This document tracked proposed changes for the v0.2 release. v0.2.0 shipped on 2026-05-02. Items remain here as a historical record of what was scoped, what landed, and what was held for v0.3. New v0.3 proposals live in [`v0.3-roadmap.md`](v0.3-roadmap.md).

Status legend: `Proposal` (drafted, not decided) | `Approved` (committed for v0.2) | `Implemented (v0.2 Phase N)` (shipped in the cited phase) | `Deferred to v0.3` (held for the next cycle)

---

## Source of these proposals

These items came out of using the v0.1 skill against a real shipping project (Stuffolio, ~50-row UNFORGET.md, App Store submission cycles) for several weeks. Most are gaps that only show up at production scale; the v0.1 spec was validated against smaller files. The header notice and format-version marker (already shipped to the public README + example file) were the immediate-safety subset of this list.

The Tier 6 items below came from a separate exercise: simulating a fresh `/unforget init` against a scratch copy of Stuffolio with the live UNFORGET.md removed. That test surfaced 9 distinct ways the v0.1 surface-survey misses real deferred work in a complex project.

## Tier 2 — Recoverability (proposals for v0.2)

| ID | Item | Status | Notes |

## Tier 3 — Format clarifications (proposals for v0.2)

| ID | Item | Status | Notes |
|---|---|---|---|
| F1 | Archive policy for closed rows at promotion time | Deferred to v0.3 | Recommended: at `promote`, move all `Fixed / Skipped` rows older than the previous release to `UNFORGET-archive-YYYY-MM.md` in the same directory. Main file stays scannable; archive file preserves history. Tradeoff: cross-references from plans/commits to archived rows have to follow the archive trail. Mitigated by archive filename being predictable. Held for v0.3 to gather beta feedback on whether the trail-following pain is real. |
| F2 | `Hidden` Status value (or convention `Fixed (workaround)`) | Deferred to v0.3 | Today, hiding a feature behind a flag and marking the row Fixed is misleading: a future reader can't tell the underlying functionality was actually fixed. Two options: add `Hidden` to the Status enum (cleanest but expands the enum), or document a convention `Fixed (workaround)` as text in the Status cell (no enum change but harder to filter on programmatically). Recommend the enum addition. Permanent vocabulary change; held for v0.3 with beta input. |
| F3 | `External` Target value for items waiting on third parties | Deferred to v0.3 | Items blocked on vendor approvals, regulatory reviews, partner integrations are common. Folding them into "deferred" loses information. Adds a 5th Target value: `⏳ EXTERNAL`. Tradeoff: vocabulary expansion is permanent. Held for v0.3 with beta input. |
| F4 | Pre-release ship gate as auto-derived view, not hand-maintained | Deferred to v0.3 | Stuffolio's UNFORGET.md hand-maintains a "Pre-release ship gate" table at the bottom that duplicates THIS-target rows from the main tables. Sync drift is invisible. v0.2's `/unforget list --target=THIS` already provides this view; spec should document it as the canonical answer to "can we ship?" and discourage hand-maintained ship-gate tables. Lower priority than other v0.2 work; v0.3 implementation. |

## Tier 4 — Feature additions (proposals for v0.2)

| ID | Item | Status | Notes |
|---|---|---|---|
| C1 | `/unforget close <ID>` subcommand | Deferred to v0.3 | Today closure is hand-edit. New command prompts for closure type (real fix / workaround / won't fix / superseded), asks if a follow-up row should be spawned, writes the closure pointer at the top of the detail block, updates Status, optionally creates the spawned row. ~5 seconds vs current "find row, find detail, edit two places, hope you don't miss the ship-gate table." Non-trivial design work; held for v0.3. |
| C2 | `/unforget delete <ID>` subcommand | Deferred to v0.3 | For rows that turned out invalid (duplicate, mistaken capture). Confirms before write, dry-run by default, retains the ID as "Retired" so cross-references don't silently point at a different row later. v0.3. |
| C3 | Spawn-link automation | Deferred to v0.3 | When `close` spawns a follow-up, both rows get reciprocal `Spawned-from:` / `Spawns:` lines automatically. `add` / `edit` / `delete` maintain the cross-references. Manual chain maintenance rots; automation prevents it. Builds on C1; v0.3. |
| C4 | `/unforget link <ID-from> <ID-to>` for retroactive chain creation | Deferred to v0.3 | If a follow-up row was created without using `close`, this command links the pair after the fact. v0.3. |

## Out of scope for v0.2

These came up during the review but aren't urgent enough or have unclear designs.

| Item | Why deferred |
|---|---|
| ID collision avoidance between AI assistants writing to UNFORGET.md concurrently | Multi-skill coordination is a future problem; rare in practice. |
| Auto-commit hook hardening (prettier / markdownlint compatibility) | Address reactively when reports come in. |
| Stale-prompt protection (continuation prompts pointing at outdated tracking files) | Session-level discipline issue, not a skill issue. |
| Per-row column visibility | Ruled out (anti-pattern). |
| Custom column reordering | Ruled out (anti-pattern). |
| Multiple-file UNFORGET.md splits | Ruled out (anti-pattern). |
