## Context

This document collects improvement suggestions for the unforget skill based on a close read of the v0.2 codebase, all roadmaps and test reports, and a real-world artifact: the Stuffolio project's live `UNFORGET.md` (336 lines, 44 active rows + 6 closed) plus the CLAUDE.md hook ("Deferred Work Index" section, lines 85–102).

**Important:** the Stuffolio "tailored" version isn't a forked SKILL.md — it's the live UNFORGET.md + the CLAUDE.md hook. Each item below therefore has **two scope lines**: one for the upstream spec change (this repo) and one for the reconciliation needed in the Stuffolio repo.

**Source material reviewed:**
- SKILL.md (907 lines / 62KB)
- README.md (354 lines)
- v0.2-roadmap.md, v0.2-release-plan.md, v0.3-roadmap.md
- examples/UNFORGET.md
- test-reports/ (all three)
- marketing/ Reddit draft
- Stuffolio's live `Documentation/Development/Deferred/UNFORGET.md`

**Five threads kept showing up:**

1. **The spec violates Anthropic's progressive-disclosure target.** SKILL.md loads fully into context on every activation; the documented target is ~5K tokens (~300 lines). At 907 lines / ~15K tokens, unforget is ~3× over. Every `/unforget list` call pays the full token tax even though listing rows is trivial, and the model re-interprets the entire prose spec on every call — which is both slow and non-deterministic across sessions. The fix is the standard pattern: a thin SKILL.md plus `reference/*.md` files loaded on demand.
2. **Deterministic operations are being asked of an LLM in prose.** Surface scanning, fuzzy dedup, path encoding, format-version checks, and backup pruning are all algorithms — but they live as natural-language descriptions the LLM re-derives on every call. They should be Python helpers under `scripts/` that the skill shells out to (the same pattern Anthropic uses for docx/pdf/xlsx). This also unblocks running unforget from non-Claude tooling.
3. **Auto-trigger is being diluted.** The frontmatter `description:` is ~280 words packing subcommand reference into the trigger field — irrelevant terms compete with words that should match user intent. Strip to 2-3 sentences.
4. **The v1 format is fragmenting in the wild — and Stuffolio is leading the fragmentation.** The Stuffolio file uses a **9-column** layout (Target inlined as a leading badge inside the Finding cell, e.g. `**🔴 THIS · ...**`) instead of upstream's 10-column. It also uses `In Progress` Status (project-local extension), a prototype `Wake-up:` field, an embedded ID Migration Table, a `/unforget fresh` subcommand reference, and a multi-promotion "Last promoted" narrative. v0.3 codifies some (T1 Wake-up, F2 Hidden, F3 External) but **not the column-count change**.
5. **Ship gates are intuition-picked, and cross-promotion fires unsolicited.** Phase 4 retest caught two critical bugs only because the author hand-walked Stuffolio (no regression bench). And the "Suggested next steps" block recommending radar-suite + bug-echo on every Fixed row will read as marketing on the second sighting — friction in a tool a user installed expecting respect.

The 15 items + smaller-fixes section below are scoped to skill-internal tooling (markdown + Claude Code subcommands + shell scripts + Python helpers) and **exclude every v0.3 roadmap item already queued** (C1–C3, F1–F4, T1, T2) plus everything that already shipped in v0.2.

## Detail per item

### 1. Tier SKILL.md — comply with Anthropic's progressive-disclosure target (~5K tokens / ~300 lines)

**What.** Anthropic's skill design guidance is progressive disclosure: SKILL.md loads fully into context every activation; the target is ~5K tokens (~300 lines). At 907 lines / ~62KB / ~15K tokens, the unforget SKILL.md is ~3× over budget. Every `/unforget list` (a trivial row-display action) pays the full token tax, and worse — the model re-interprets the entire spec from prose on every invocation, which is both slow and non-deterministic. The fix is the standard progressive-disclosure structure:

- **`SKILL.md` (≤300 lines):** frontmatter, four-section format at-a-glance, subcommand surface table, decision flowchart for "which subcommand do I run", pointer to companion files.
- **`reference/format.md`:** column definitions, Target/Status/Urgency enums, format-version contract, anti-patterns. Loaded only when the model is about to write or validate.
- **`reference/init.md`:** the seven-phase walkthrough (lines 138–486 of current SKILL.md). Loaded only on `/unforget init`.
- **`reference/surfaces.md`:** the six surface-survey heuristics (radar-suite ledger, audit reports, plan files, memory, code comments, GH issues). Loaded only on `init` or `import`.
- **`reference/promotion.md`:** promote ritual, dry-run mechanics, backup policy. Loaded only on `/unforget promote`.

The pattern matches how Anthropic's own example skills (artifacts, code-execution, etc.) are structured: a thin SKILL.md that names companion files, and the model `Read`s them on demand.

**Why this is a hard correctness issue, not just ergonomics.** Three concrete failure modes:

1. **Token tax on cheap operations.** Listing 5 rows requires loading 15K tokens of spec. Repeated cost across a session adds up.
2. **Re-interpretation non-determinism.** The model re-parses 907 lines of prose every call. Different sessions can reach different conclusions about edge cases (e.g., "is `In Progress` a valid Status?") because the prose has internal tensions (line 87 lists it; line 341 uses it ambiguously). Reference files broken out by concern make each decision *local*.
3. **Spec rot is invisible.** Lines 59–136 (Format) and 817–836 (How to use this file) say similar things slightly differently — exactly the kind of drift that long files accumulate. Splitting collapses each topic to one home.

**Why.** This is the strongest single argument for any item in this document: Anthropic's own design guidance is being violated, the violation has measurable cost on every call, and the fix is mechanical (move sections into companion files).

**Compatibility.** Additive at the spec level (no format-version bump). Repo layout changes — `reference/` directory gets added — but the plugin manifest and SKILL.md frontmatter are unchanged.
**Order.** Independent. Should land first — it changes the substrate every other item edits.
**Stuffolio-side reconciliation.** None. Stuffolio reads SKILL.md transparently via the plugin install; the split is invisible to the project.

### 11. `scripts/` directory — move deterministic ops out of prose into Python helpers

**What.** Surface scanning, fuzzy dedup, path encoding, format-version checks, and backup pruning are all deterministic operations currently being asked of an LLM in prose. Each call re-derives the algorithm from natural-language description, which is slow, expensive, and non-deterministic. Move them to Python helpers the skill shells out to:

- `scripts/scan_surfaces.py --root <path>` → JSON of discovered deferral artifacts
- `scripts/encode_project_path.py <abs-path>` → Claude Code memory-dir encoding
- `scripts/dedup_findings.py --candidates <file.json>` → fuzzy-merged candidate list
- `scripts/check_format_version.py <unforget.md>` → version + structural validation
- `scripts/prune_backups.py --keep 5` → backup rotation

SKILL.md then says e.g. *"To scan surfaces, run `python scripts/scan_surfaces.py --root .` and parse the JSON result."* This matches how Anthropic's own docx/pdf/xlsx skills are built (deterministic logic in scripts; the LLM orchestrates).

**Why.** Three benefits: (a) **determinism** — fuzzy-match a row across surfaces always returns the same answer regardless of session; (b) **token cost collapses** — a 4KB prose surface-scan algorithm becomes a 1-line shell call; (c) **portability** — the skill becomes runnable from non-Claude tooling (CI, GitHub Actions, cron) without re-implementing the LLM-prose logic. This is a strong candidate for the *single highest-impact change* alongside #1 — possibly the highest, since it removes the largest chunk of prose driving the SKILL.md page count.

**Compatibility.** Additive. SKILL.md prose can keep a fallback algorithm description in `reference/*.md` for environments without Python, but the canonical implementation is the script.
**Order.** Independent of #0. Pairs naturally with #1 (the prose that gets removed in #1 has its logic preserved in `scripts/` rather than lost).
**Stuffolio-side reconciliation.** None. Plugin install gains a `scripts/` dir; nothing in the Stuffolio repo changes. Indirect benefit: the bugs that took two retest cycles to find (T-8 path encoding, T-12 filename pattern, T-13 ID extraction) are exactly the kind of bug that a unit-tested Python script catches at write time.

### 0. Resolve the column-count divergence

**What.** Stuffolio's UNFORGET.md uses **9 columns** with Target inlined as a leading badge inside the Finding cell (e.g. `**🔴 THIS · Apple Wallet Pass broken promise**`). Upstream's spec mandates **10 columns** with Target as a dedicated column. Pick one canonical form upstream:

- **Option A — adopt Stuffolio's 9-column form upstream.** Rationale per Stuffolio CLAUDE.md line 94: `grep '🔴 THIS'` still finds ship-blockers, the saved column gives the Finding cell more room before hitting the 50-char limit, and Stuffolio is the only project with real-world stress data.
- **Option B — keep upstream 10-column, treat 9-column as a documented preset.** Add a "Compact" preset alongside Standard / Lean / Continuous (SKILL.md line 131). Stuffolio's file remains conformant under the Compact preset.
- **Option C — keep both, declare 9-column non-conformant, migrate Stuffolio to 10-column** at the next promote. Simplest spec, biggest churn.

**Why this gates other items.** Without resolving this, items #2 (`migrate`), #3 (test corpus), #4 (`doctor`), and any v0.3 validator will all flag Stuffolio's file as malformed even though it was authored by the skill author and works in production. Every other item touching format validation is downstream of which column model is canonical.

**Recommendation.** Option B (Compact preset). Lowest churn, lets Stuffolio stay valid as-is, and gives other adopters a sanctioned shorter format. The test corpus (#3) can ship golden fixtures for both Standard and Compact.

**Compatibility.** Option B is purely additive at the spec level (no version bump). Options A and C both touch v1 → v2.

**Order.** Independent, but it gates #2, #3, #4. Land before those.

**Stuffolio-side reconciliation.**
- Option B: no change required. Stuffolio CLAUDE.md line 94 already documents the inlined-Target convention; just add a one-line cross-reference to the upstream "Compact" preset name.
- Option A: rewrite upstream spec; Stuffolio file unchanged.
- Option C: `/unforget migrate` rewrites all 44 active rows + 6 closed rows back to 10-column; CLAUDE.md line 94 paragraph deletes.

### 12. Strip frontmatter description to 2-3 sentences

**What.** SKILL.md frontmatter `description:` is currently ~280 words covering activation triggers, file format, four-section structure, and a full subcommand list. Strip to 2-3 sentences:

```yaml
description: |
  A single source of truth for deferred work — paused plans, mid-task spillover,
  audit findings, and observed bugs — kept in one UNFORGET.md per project so
```

**Why.** Skills auto-trigger on description-embedding match. Packing 280 words of subcommand reference into the trigger field dilutes the activation signal — irrelevant terms (column names, status enum values, phase numbers) compete with the words that should match user intent. A focused 2-3 sentence description matches user prompts more precisely.

**Compatibility.** Frontmatter-only edit. No format-version impact, no behavior change at activation time.
**Order.** Independent. Pairs with #1.
**Stuffolio-side reconciliation.** None.

### 13. Gate cross-skill cross-promotion behind a marker or `--sweep` flag

**What.** The "Suggested next steps" block at SKILL.md lines 525–539 fires every time a row is closed with `/unforget edit --status=Fixed`. Even with the detect-then-recommend logic at lines 540–547, the same recommendation reading on the second / fifth / twentieth Fixed row will read as marketing. Two options:

- **Option A (recommended):** Once-per-project marker. Write `<!-- unforget-cross-promo-shown: 2026-05-04 -->` to the file (or sidecar) the first time the recommendation fires. Subsequent closures suppress it. Re-show only on `/unforget --reset-prompts` or after a configurable cooldown (90 days?).
- **Option B:** Move recommendation behind an opt-in `--sweep` flag on `edit`. Default closure prints "Closed. Run `/unforget edit P3 --sweep` to also run radar-suite + bug-echo on this fix."

**Why.** Users who installed unforget to track deferred work didn't sign up to be re-pitched on radar-suite + bug-echo on every closure. Repeated unsolicited pitches in a tool a user *paid attention to install* are exactly how skill-suite ecosystems erode user trust. Once is helpful; second time is friction; tenth time is noise.

**Compatibility.** Additive. The marker approach uses an HTML comment; the flag approach is a CLI surface change.
**Order.** Independent.
**Stuffolio-side reconciliation.** Stuffolio's UNFORGET.md gets a one-time HTML-comment marker the first time a row is closed after this lands. No further edits.

### 15. "When to use unforget vs an existing tracker" section

**What.** Add a section to the README addressing the obvious objection: "I already use Jira / Linear / GitHub Projects — why would I add another tracker?" Short, honest answer covering:

- **Use unforget when:** you are a solo / small-team developer; deferred items currently scatter across Deferred.md, plan files, memory files, audit ledgers; you want a tracker that the AI can read alongside your code; you don't have a separate PM tool.
- **Use Jira / Linear / GitHub Projects when:** you have a separate non-developer audience (PMs, designers); you need ticket assignment, sprints, story points, custom workflows; the tracker is your team's source of truth for *all* work, not just deferrals.
- **Coexistence pattern:** UNFORGET.md is for *code-adjacent technical debt with release-cycle commitment*; the external tracker is for cross-functional work. Cross-link as needed (UNFORGET.md row points at JIRA-1234).

**Why.** The Anti-patterns section forbids splitting *within* unforget but says nothing about teams who already have a canonical tracker. Without this section, the very-real "we already have Linear" objection has no counter, and the obvious answer (use them for different things) is unstated.
